<objective>
Wire the four locked server-side emits into the existing route handlers, immediately after the successful `pool.query` that produces the state change. This phase delivers the actual realtime fan-out that REQ-08 demands — without these emits, Plan 03's frontend listeners would have nothing to listen to.

Scope of changes (every emit lives inline next to the SQL that triggered it; no event bus, per CONTEXT.md locked decision and CLAUDE.md L57):
1. `donation.routes.js` POST `/` → `donation:available` to `role:NGO`.
2. `request.routes.js` POST `/` → `request:incoming` to `user:{donor_id}`.
3. `request.routes.js` PUT `/:id` (donor approve/reject) → `request:resolved` to `user:{ngo_id}`, NO otp.
4. `request.routes.js` PUT `/:id/volunteer` (NGO assigns volunteer + OTP issued) → `request:resolved` to `user:{ngo_id}`, WITH otp.
5. `request.routes.js` POST `/:id/verify-otp` → `pickup:collected` to BOTH `user:{donor_id}` AND `user:{ngo_id}`.

Plus one mechanical SQL fix in (5): the verify-otp SELECT must add `d.donor_id` to its projection so the emit can fan to the donor's room.

Purpose: Realize the four locked event names with payloads exactly as specified in CONTEXT.md, with room targeting that prevents the OTP-leakage threat (T-08-04 / Pitfall 7).
Output: All four event types are emitted from the server, each scoped to the minimum necessary rooms.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated REST request → Socket.IO emit | The emit fans the state change to other connected users. Anything in the payload becomes visible to every socket in the targeted room(s). |
| Per-user room (`user:{id}`) → Per-role room (`role:NGO`) | Per-user is one user; per-role is many. Sending sensitive data to a role room is a multi-user disclosure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-03 | Information Disclosure | `donation:available` payload could carry donor PII (donor name, address) into the role:NGO broadcast | accept | The payload is `{ donation }` where `donation` is the row from `INSERT INTO donations RETURNING *`. The schema (`server/config/schema.sql`) has `address` (the pickup address — by definition shared with all NGOs that might claim it) and a foreign-key `donor_id` (an integer, not PII). Donor name is NOT in the donations row — NGO-side enrichment happens via existing `JOIN users` on `GET /donations/available`. The payload here is a strict subset of what `GET /donations/available` already returns to every NGO. No new disclosure. |
| T-08-04 | Information Disclosure | `request:resolved` carrying `otp` accidentally targeting `role:NGO` instead of `user:{ngo_id}` (Pitfall 7 — the trust-backbone risk) | mitigate | (a) Both `request:resolved` emits target `user:${...}` per-user rooms — verified by acceptance_criteria grep `grep -E "to\(['\"]role:NGO['\"]\)\.emit\(['\"]request:resolved"` returning NO matches. (b) Code-review checklist embedded in this plan: any future emit whose payload may carry `otp` MUST target a `user:{id}` room. (c) The locked decision says no client-emitted events — clients cannot self-join arbitrary rooms (re-asserted by Plan 01's `server/socket.js` design). |
| T-08-05 | Information Disclosure | `request:incoming` payload includes `donation` row — could leak donor info to an NGO that hasn't been approved | accept | The donor row in `donation.rows[0]` (used in the request:incoming emit) is the donation row, not the user row. It contains the same fields already visible in `GET /donations/available` to all NGOs. The NGO that just submitted the request triggered this emit, and the recipient is the donor (the donor's own data — not new disclosure). |
| T-08-06 | Information Disclosure | `pickup:collected` payload includes the OTP-verified request (which still has `request.otp` populated post-verify) | accept | Per RESEARCH.md threat model: post-verification, the OTP is no longer secret — its single use happened. Even so, the emit only goes to the donor (who entered the OTP themselves) and the assigned NGO (who generated it). No new disclosure surface. |
| T-08-07 | Tampering | A donor approves a request but the emit fails silently (e.g., io is undefined) — UI never updates, donor refreshes and sees stale state | accept | `req.app.get('io')` is set up in Plan 01's `app.set('io', io)`. If app.set is missing, `req.app.get('io')` returns `undefined` and the emit throws synchronously. That throw bubbles to the existing `try/catch` and returns 500 to the donor — they retry. The risk is operational (broken deploy) not security. Mitigated by Plan 01's acceptance test for `app.set('io', io)`. |
| T-08-13 | DoS | A malicious or buggy NGO spam-creates donations → flood of `donation:available` emits to every connected NGO | accept | Donation creation is gated by `requireRole('DONOR')` plus the donation creator must be `is_verified=true` (admin gate). Pre-pilot scale: not a concern. Post-pilot: rate-limit at the REST layer in P11. |
| T-08-14 | Repudiation | A user denies receiving the realtime event ("I never saw the OTP") | accept | Out of scope for P8. Logging of socket events lands in P13 (structured logs). For pilot, the OTP also persists in `requests.otp` and is reachable via `GET /api/requests/mine` — REST-level audit trail is unchanged. |
