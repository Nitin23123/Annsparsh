<objective>
Wire the Socket.IO client into both dashboards so REQ-08 success criteria 1-5 are met end-to-end. Three changes:

1. `src/socket.js` — switch from object-form `auth: { token }` (evaluated once at module load) to function-form `auth: (cb) => cb({ token: localStorage.getItem('token') || '' })` (re-evaluated on every handshake). This fixes Pitfall 1 (stale token on reconnect).

2. `src/components/DonorDashboard.jsx` — register listeners for `request:incoming` (new request lands) and `pickup:collected` (donation flips to COLLECTED), plus a `connect` handler that triggers `fetchData()` for the on-reconnect refresh (locked decision: one-shot REST refetch, no replay buffer). All listeners registered in a dedicated `useEffect` with named-handler cleanup (Pitfall 3 + 4).

3. `src/components/NGODashboard.jsx` — register listeners for `donation:available`, `request:resolved` (handles BOTH the approve-emit and the volunteer-assign-emit by merging `otp ?? r.otp`), and `pickup:collected`, plus the same on-`connect` refetch. Same named-handler cleanup pattern.

Purpose: Without these wires, Plan 02's server emits go to the void on the client side. This plan delivers all five REQ-08 success criteria as observable UI behavior.
Output: Both dashboards reflect every state change in real time without manual refresh, survive tab close/reopen, and never duplicate toasts in dev StrictMode.

Reconnect-window note (researcher Open Question #4 resolution): REQ-08's 2-second budget applies to STEADY-STATE event delivery. During an active reconnect attempt, Socket.IO's default 1s backoff with 0.5 jitter and 5s max means the worst case is up to 5s before the `connect` event fires and the on-`connect` REST refetch recovers missed state. This is acceptable for v1; if pilot users report >5s recovery, add a manual reconnect button or shorten `reconnectionDelayMax` to 2000.
</objective>

import { io } from 'socket.io-client';

const SOCKET_URL = (import.meta.env.VITE_API_URL || 'http://localhost:5000/api').replace('/api', '');

const socket = io(SOCKET_URL, {
    transports: ['websocket'],
    autoConnect: true,
    auth: (cb) => cb({ token: localStorage.getItem('token') || '' }),
    withCredentials: true,
});

socket.on('connect_error', (err) => {
    if (err.message === 'Invalid token' || err.message === 'No token' || err.message === 'Not verified') {
        return;
    }
});

export default socket;

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage -> Socket handshake | The JWT in `localStorage.token` is the only authentication signal sent on each handshake. If the storage is compromised (XSS), the socket inherits the same auth as REST. |
| Singleton socket -> Multiple component listeners | `src/socket.js` exports a module-level singleton. Multiple components register listeners on it; sloppy cleanup (`socket.off('event')` without handler arg) tears down listeners other components own. |
| `connect_error` event -> User session state | An auth error on the socket (`'Invalid token'`, `'Not verified'`) does NOT itself clear localStorage or redirect. Only a REST 401 does (via `src/api.js:24-34`). This separation prevents a flapping socket from forcing a logout. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

Pitfall 1 + State-of-the-Art note in RESEARCH.md. |
