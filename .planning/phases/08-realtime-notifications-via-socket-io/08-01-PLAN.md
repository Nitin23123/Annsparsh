Create a new file `server/socket.js` with exactly this content (CommonJS, no comments — per CLAUDE.md line 61, the `why` is documented in this plan, not in code):

```javascript
const { Server } = require('socket.io');
const jwt = require('jsonwebtoken');
const pool = require('./config/db');

module.exports = function buildSocketServer(httpServer) {
    const io = new Server(httpServer, {
        cors: {
            origin: ['http://localhost:5173', 'http://localhost:5174'],
            credentials: true,
        },
    });

    io.use(async (socket, next) => {
        const token = socket.handshake.auth?.token;
        if (!token) return next(new Error('No token'));
        try {
            const decoded = jwt.verify(token, process.env.JWT_SECRET);
            const r = await pool.query('SELECT is_verified FROM users WHERE id = $1', [decoded.id]);
            if (!r.rows.length || !r.rows[0].is_verified) return next(new Error('Not verified'));
            socket.user = decoded;
            next();
        } catch {
            next(new Error('Invalid token'));
        }
    });

    io.on('connection', (socket) => {
        socket.join(`user:${socket.user.id}`);
        socket.join(`role:${socket.user.role}`);
    });

    return io;
};
```

Hard rules (do NOT deviate):
1. CORS origin array MUST be `['http://localhost:5173', 'http://localhost:5174']` — exact same string literals as `server/app.js:7`. If those two lists drift, the polling-fallback handshake will fail (Pitfall 6 in RESEARCH.md).
2. The error messages MUST be the exact strings `'No token'`, `'Invalid token'`, and `'Not verified'`. Frontend (Plan 03) checks for `'Invalid token'` and `'No token'` in `connect_error.message`.
3. Reject unverified users in `io.use()` — this resolves the researcher's Open Question #2 with disposition: enforce, defense in depth (CLAUDE.md line 77 says do not bypass is_verified).
4. Use `socket.handshake.auth?.token` — NOT `socket.handshake.query.token` and NOT the `Authorization` header. The frontend will pass it via the `auth` option (locked decision).
5. Room names: `user:{id}` and `role:{ROLE}` — exactly. The route emits in Plan 02 hardcode these formats.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Untrusted browser → Socket.IO handshake | Any client on the public internet can attempt a WebSocket upgrade. The JWT in `auth.token` is the only authentication signal. |
| Socket.IO io.use() middleware → application rooms | After this gate, the socket is "trusted" — server-side code joins it to rooms based on the decoded JWT claims. Anything past this point assumes the user identity in `socket.user` is correct. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Spoofing | Socket handshake (`server/socket.js` io.use) | mitigate | Reject sockets without `auth.token` (next(new Error('No token'))). Verify the token with the same `jwt.verify(token, process.env.JWT_SECRET)` shape used by `server/middleware/auth.js:8`. Reject any signature/claim failure with `next(new Error('Invalid token'))`. |
| T-08-02 | Tampering | Forged or modified JWT | mitigate | HMAC verification inside `jwt.verify` is the same trust source as REST. No additional mitigation in this phase — risk is identical to existing REST risk. The per-developer-env `JWT_SECRET` (STATE.md line 55) is a known operational risk tracked there, not in this plan. |
| T-08-08 | Elevation of Privilege | A user with `is_verified=false` connecting and joining `role:NGO` to receive donations before admin verification | mitigate | After `jwt.verify` succeeds, run `SELECT is_verified FROM users WHERE id = $1` and reject with `next(new Error('Not verified'))` if false or row missing. Resolves researcher's Open Question #2 with the more secure default. Cost: one DB query per socket connection — negligible at pilot scale. |
| T-08-09 | Information Disclosure | CORS misconfiguration (`origin: '*'` with `credentials: true`) lets attacker.com initiate sockets in a victim's authenticated session | mitigate | Pin `cors.origin` to the same allow-list as REST (`['http://localhost:5173', 'http://localhost:5174']`). No `'*'`. Production origin must be added to BOTH `server/app.js` REST CORS AND `server/socket.js` Socket.IO CORS in the same commit (Pitfall 6 in RESEARCH.md). |
| T-08-10 | Information Disclosure | Stray `app.listen()` left after the refactor — REST works, sockets silently never connect, no error in logs (developers think nothing is wrong) | accept | Not a security threat per se, but the failure mode hides successful auth bypass attempts. Mitigation: Task 3 acceptance criteria explicitly checks for stray `app.listen`. Not a STRIDE-tracked threat after the check. |
| T-08-11 | Spoofing | Client tries to self-join arbitrary room (e.g., `socket.emit('join', 'user:42')`) | mitigate | No client→server event handlers are registered in `server/socket.js`. Only `io.on('connection')` joins rooms, and only based on values from the verified JWT. Locked decision: sockets are listen-only client-side. |
| T-08-12 | DoS | Reconnect storm from a single misbehaving client triggers repeated `jwt.verify` + DB lookup | accept | Default Socket.IO client reconnection delay is 1s with 0.5 jitter, max 5s. Per-connection cost is one `jwt.verify` (fast, in-memory) + one indexed `SELECT is_verified` (sub-millisecond at pilot scale). Rate-limiting at the io.use() layer is deferred to P11 alongside the Express rate-limit work. |
</threat_model>
